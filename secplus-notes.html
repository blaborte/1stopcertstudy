1.0 General Security Concepts 12%
🛡️ Security Control Types & Categories ▼
Security controls are the mechanisms used to reduce risk. Every SY0-701 scenario question asks you to identify or select a control — knowing the taxonomy perfectly is non-negotiable.
Control Categories
  • Technical (Logical): implemented via technology — firewalls, encryption, MFA, ACLs, IDS/IPS, antivirus
  • Managerial (Administrative): policies, procedures, risk assessments, security awareness training, background checks
  • Operational: day-to-day procedures carried out by people — security guards, log reviews, incident response drills
  • Physical: protect physical assets — locks, cameras, biometrics, fences, bollards, mantraps, lighting
Control Types (by function)
  • Preventive: stops attacks before they occur — firewall, encryption, access control, training
  • Deterrent: discourages attacks — warning signs, visible cameras, security lighting, guards
  • Detective: identifies attacks in progress or after — IDS, SIEM, logs, motion sensors, CCTV
  • Corrective: restores systems after an incident — patches, backups, DRP, antivirus quarantine
  • Compensating: alternative control when primary isn't feasible — additional monitoring instead of MFA
  • Directive: guides behavior — policies, procedures, training, acceptable use agreements
🎯
Key exam pattern: CCTV = deterrent + detective. Firewall = preventive + technical. Patches = corrective. Guards = physical + operational. Any single control can satisfy multiple categories simultaneously.
🔐 Cryptography Deep Dive ▼
Cryptography is one of the heaviest SY0-701 topics. Know every algorithm, its key size, its use case, and its vulnerabilities. Cryptography questions appear throughout all five domains.
Symmetric Algorithms
  • AES (Advanced Encryption Standard): 128/192/256-bit — current gold standard. Block cipher. Used for data at rest and bulk data.
  • 3DES: applies DES three times — 168-bit effective — legacy, being phased out
  • Blowfish/Twofish: fast alternatives to AES — less common but still used
  • ChaCha20: stream cipher — used in TLS 1.3 and mobile devices where AES hardware acceleration unavailable
Asymmetric Algorithms
  • RSA: 2048+ bit keys required for security — based on prime factorization difficulty
  • ECC (Elliptic Curve Cryptography): equivalent security with much smaller keys (256-bit ECC ≈ 3072-bit RSA) — preferred for mobile
  • Diffie-Hellman: key EXCHANGE protocol — allows two parties to establish shared secret over insecure channel without sending the key
  • DHE/ECDHE: ephemeral variants — provide Perfect Forward Secrecy (PFS)
Hashing Algorithms

PKI & Certificate Concepts
  • CA (Certificate Authority): issues and signs digital certificates — root CA → intermediate CA → end-entity cert
  • Certificate contents: subject name, public key, validity period, issuer, digital signature, SAN (Subject Alternative Name)
  • CRL (Certificate Revocation List): list of revoked certificates — checked before trusting a cert
  • OCSP (Online Certificate Status Protocol): real-time certificate status check — better than CRL
  • Self-signed certificate: signed by itself, not a trusted CA — generates browser warnings
  • Wildcard certificate: *.domain.com — covers all subdomains
  • Code signing certificate: used to sign software and scripts — proves origin and integrity
🔄 Change Management & Zero Trust ▼
Change management and Zero Trust Architecture are newer heavy topics in SY0-701 that reflect modern enterprise security reality.
Change Management in Security
  • Approval process: RFC → risk assessment → CAB approval → maintenance window → rollback plan → implement → verify
  • Security impact: every change can introduce new vulnerabilities — patch one hole, open another
  • Version control: track changes to code, configurations, and policies — rollback capability
  • Configuration drift: systems gradually diverge from secure baseline — continuous monitoring catches this
  • IaC (Infrastructure as Code): define infrastructure in code — consistent, auditable, version-controlled deployments
Zero Trust Architecture (ZTA)
  • Core principle: "Never trust, always verify" — no implicit trust based on network location
  • Policy-based authentication: every access request evaluated against current policy — not just who you are but also device health, location, behavior
  • Microsegmentation: network divided into tiny segments — lateral movement is severely limited
  • Least privilege access: minimum permissions at every level — JIT (Just-in-Time) access for privileged accounts
  • Continuous validation: trust is never permanent — re-validated constantly during session
🔑
Zero Trust vs traditional perimeter: Traditional = hard outside, soft inside. Zero Trust = verify everything everywhere, always. A compromised insider account gets no special trust in ZTA.
2.0 Threats, Vulnerabilities & Mitigations 22%
🎭 Threat Actors & Motivations ▼
Understanding who is attacking you and why shapes every defensive decision. SY0-701 heavily tests threat actor classification and their typical tools, techniques, and procedures (TTPs).
Threat Actor Types
Nation-State
Most sophisticated — APT groups, massive resources, espionage/sabotage. Examples: Lazarus Group, APT28/Fancy Bear.
Hacktivist
Ideologically motivated — disrupt or embarrass targets. Anonymous, hacktivists target organizations they oppose.
Cybercriminal
Financial motivation — ransomware, data theft, fraud, dark web sales. Most common threat type.
Insider Threat
Current or former employee/contractor with privileged access — most dangerous due to existing access.
Script Kiddie
Low skill — uses pre-made tools without understanding them. Opportunistic, high volume, low sophistication.
Competitor
Corporate espionage — steal IP, sabotage operations. Harder to detect, motivated by market advantage.
Attack Surfaces & Threat Vectors
  • Message-based: phishing (email), smishing (SMS), vishing (voice) — largest attack vector
  • File-based: malicious PDFs, Office macros with embedded code, executables
  • Web-based: drive-by downloads, malicious ads (malvertising), browser exploits
  • Removable media: USB drops — attacker leaves infected USB in target's parking lot
  • Supply chain: compromising software or hardware before it reaches the target — SolarWinds attack
  • Unsecured networks: evil twin APs, unencrypted Wi-Fi, open ports, default credentials
  • Unpatched systems: known CVEs with public exploits — patch management prevents this vector
🦠 Malware Types & Attack Indicators ▼
Malware analysis is a core SY0-701 competency. You need to identify malware by behavior, understand propagation methods, and recognize indicators of compromise.
Malware Categories
  • Ransomware: encrypts data/systems → demands payment. Double-extortion = steal + encrypt. Triple-extortion = also DDoS.
  • Trojan: disguised as legitimate software. RAT = Remote Access Trojan — provides full remote control.
  • Worm: self-replicates across network without human action. Exploits known vulnerabilities automatically.
  • Virus: requires host file and user action to spread. Attaches to programs or boot sectors.
  • Rootkit: kernel-level hiding. Intercepts OS calls. Survives reboots. Extremely difficult to detect/remove.
  • Spyware: covertly monitors user activity. Keylogger subset captures every keystroke.
  • Adware: displays advertisements. Often bundled with free software.
  • Botnet/C2: network of compromised devices controlled by attacker via Command and Control server.
  • Logic bomb: triggers on specific condition (date, event, employee termination).
Indicators of Compromise (IoCs)
  • Unusual outbound traffic: large data exfiltration, traffic to known-malicious IPs/domains
  • New unauthorized admin accounts: attacker created persistence through privileged account
  • Disabled security tools: malware often disables AV, EDR, firewalls before executing payload
  • Lateral movement: internal scanning, credential dumping (Mimikatz), RDP between servers
  • Encrypted C2 traffic: malware using HTTPS or DNS tunneling to evade detection
  • Anomalous login times: accounts accessing systems at unusual hours or from unusual locations
🔓 Vulnerability Types & Mitigations ▼
Vulnerabilities are the weaknesses that attackers exploit. SY0-701 tests your ability to categorize vulnerabilities and select appropriate mitigations for each type.
Application Vulnerabilities
  • SQL Injection (SQLi): malicious SQL in input fields — accesses, modifies, or destroys database. Fix: parameterized queries, input validation.
  • Cross-Site Scripting (XSS): injects JavaScript into web pages via poor input validation — steals cookies/sessions. Fix: output encoding, CSP headers.
  • Cross-Site Request Forgery (CSRF): tricks authenticated user's browser into making unintended requests. Fix: CSRF tokens, SameSite cookies.
  • Buffer overflow: writes beyond allocated memory — can execute arbitrary code. Fix: bounds checking, DEP, ASLR.
  • Race condition: exploits timing between check and use of a resource. Fix: atomic operations, proper locking.
  • Insecure direct object reference: exposes internal implementation objects directly in URLs. Fix: access control checks.
Infrastructure Vulnerabilities
  • Unpatched systems: known CVEs with public exploits — most easily preventable category
  • Default credentials: unchanged factory passwords — first thing attackers try
  • Misconfiguration: overly permissive firewall rules, exposed admin interfaces, debug mode in production
  • Weak cryptography: MD5/SHA-1, DES, RC4, small RSA keys — upgrade to modern algorithms
  • Zero-day: unknown vulnerability with no patch — requires anomaly detection and zero trust
  • Supply chain vulnerability: compromised component before delivery — code signing, trusted vendor programs
🎯
Exam focus: XSS and SQLi both exploit input validation failures. The key difference: SQLi targets the DATABASE; XSS targets OTHER USERS's browsers.
3.0 Security Architecture 18%
🏛️ Security Architecture Models ▼
Architecture is about how security is designed into systems, not bolted on afterward. SY0-701 tests cloud architectures, network segmentation, and resilience design.
Cloud Security Architecture
  • Shared Responsibility Model: security is divided between provider and customer — the line shifts by service model
  • IaaS: provider secures hypervisor/hardware. Customer secures OS, apps, data, networks.
  • PaaS: provider secures hardware + OS + runtime. Customer secures apps and data.
  • SaaS: provider secures nearly everything. Customer secures user access and data configuration.
  • Serverless / Functions: provider manages all infrastructure. Customer secures function code and invocation permissions.
  • Containers: customer responsible for container images, configuration, and orchestration security.
Network Segmentation
  • DMZ (Screened subnet): public-facing servers isolated between two firewalls — web, email, DNS
  • VLANs: logical segmentation — separate guest, IoT, BYOD, production, management traffic
  • Microsegmentation: fine-grained isolation of individual workloads — limits lateral movement in breach
  • Air gap: physical isolation — no network connection possible. Used for critical infrastructure (SCADA, ICS)
  • IoT/OT segmentation: smart devices and operational technology must be isolated from corporate network
Resilience & Recovery Design
  • 3-2-1 backup rule: 3 copies, 2 different media types, 1 offsite — minimum for disaster recovery
  • Geographic redundancy: multiple data centers in different locations — protects against regional disasters
  • Load balancing: distributes traffic — performance + fault tolerance
  • Clustering: multiple servers appearing as one — automatic failover when node fails
  • Replication: synchronous (zero data loss, performance hit) vs asynchronous (some data loss risk, better performance)
🛡️ Data Protection Strategies ▼
Data protection spans the entire data lifecycle. SY0-701 tests encryption choices, DLP, classification, and secure disposal.
Encryption Strategy
  • Data at rest: AES-256 FDE (Full Disk Encryption) — BitLocker (Windows), FileVault (Mac), LUKS (Linux)
  • Database encryption: TDE (Transparent Data Encryption) — encrypts data files on disk
  • Data in transit: TLS 1.2/1.3 minimum — disable SSLv3, TLS 1.0, TLS 1.1
  • End-to-end encryption: only sender and recipient can read — even service provider cannot decrypt
  • Key management: HSM (Hardware Security Module) — tamper-resistant key storage and crypto operations
Data Loss Prevention (DLP)
  • Network DLP: monitors outbound traffic — blocks SSNs, credit card numbers, classified data from leaving network
  • Endpoint DLP: controls data on devices — blocks USB writes, unauthorized cloud uploads, personal email
  • Cloud DLP: monitors and controls data in cloud storage and SaaS applications
  • Classification triggers DLP: labels on documents determine which DLP rules apply
Secure Data Disposal
  • Degaussing: magnetic field erases magnetic media — tapes and spinning HDDs only, not SSDs
  • Shredding/Disintegration: physical destruction — the only reliable method for SSDs
  • Cryptographic erasure: destroy encryption key — data remains but is unreadable (fast for cloud storage)
  • Overwriting: write random patterns over data multiple times — effective for HDDs
  • Data remanence: residual data after deletion — reason secure disposal is critical
4.0 Security Operations 28%
🔧 Security Operations & Monitoring ▼
Security operations is where theory meets reality. SY0-701 emphasizes SIEM, EDR, SOAR, and the investigative skills needed to detect and respond to incidents.
Security Monitoring Stack
SIEM
Aggregates and correlates logs from across the environment. Provides alerting, dashboards, forensic analysis, retention. Does NOT encrypt logs.
SOAR
Security Orchestration, Automation, and Response — automates response playbooks triggered by SIEM alerts. Reduces mean time to respond.
EDR
Endpoint Detection and Response — continuous monitoring of endpoint behavior. Detects, investigates, and responds at the device level.
XDR
Extended Detection and Response — EDR + network + cloud + identity. Cross-layer correlation for complex attacks.
NDR
Network Detection and Response — behavioral analytics on network traffic. Detects lateral movement and exfiltration.
IDS/IPS
IDS = passive detection + alert. IPS = inline detection + automatic block. NIDS via port mirror/TAP.
Vulnerability Management Lifecycle
  • 1. Discover: inventory all assets — you cannot protect what you do not know exists
  • 2. Prioritize: risk-based — CVSS score, exploitability (EPSS), asset criticality
  • 3. Assess: credentialed scans find more (authenticated to system) vs non-credentialed (external view)
  • 4. Remediate: patch, mitigate, or accept with documentation
  • 5. Verify: rescan to confirm remediation worked
  • False positive: scanner reports vulnerability that doesn't exist — waste of remediation time
  • False negative: real vulnerability not detected — dangerous gap in visibility
🚨 Incident Response & Digital Forensics ▼
IR and forensics are heavily tested in SY0-701. Know the phases, evidence handling requirements, and what tools are used at each stage.
IR Phases (NIST/CompTIA)
  • 1. Preparation: policies, tools, playbooks, team training — before anything happens
  • 2. Identification / Detection: confirm incident occurred — SIEM alerts, IoCs, anomaly detection
  • 3. Containment: short-term (isolate affected systems) and long-term (patch, credential rotation)
  • 4. Eradication: remove root cause — malware, attacker persistence mechanisms, compromised accounts
  • 5. Recovery: restore systems from known-good backups, verify functionality, monitor closely
  • 6. Lessons Learned: post-incident review, update playbooks, report to management
Digital Forensics Principles
  • Chain of custody: documented record of every person who handled evidence — critical for legal proceedings
  • Order of volatility: collect most volatile data first — CPU registers/cache → RAM → swap → disk → logs → archives
  • Write blockers: hardware/software preventing modifications during forensic acquisition
  • Legal hold: freeze deletion of potentially relevant data — triggered by litigation or investigation
  • Evidence integrity: hash evidence before and after — MD5/SHA-256 hashes prove no tampering
🔑
Critical forensics rule: Never work on original evidence — always work on forensic copies. The original must be preserved unchanged with documented chain of custody.
🔑 Identity & Access Management ▼
IAM is one of the largest SY0-701 topic areas. Modern identity spans cloud federation, PAM, and zero trust verification.
Authentication Protocols & Standards
  • SAML 2.0: XML-based SSO federation — enterprise SSO between organizations. SAML assertions carry authentication info.
  • OAuth 2.0: authorization framework — allows third-party apps to access resources without sharing credentials. "Login with Google."
  • OIDC (OpenID Connect): authentication layer on top of OAuth 2.0 — provides identity (who you are) not just authorization (what you can do)
  • RADIUS: UDP, AAA for network access — Wi-Fi, VPN, 802.1X
  • LDAP/Active Directory: central directory for users, groups, computers — authentication backend
  • FIDO2/WebAuthn: passwordless authentication — hardware keys (YubiKey), biometrics
Privileged Access Management (PAM)
  • PAM: controls, monitors, and audits privileged account access — separates privileged and non-privileged accounts
  • PIM (Privileged Identity Management): manages privilege elevation — JIT access that expires
  • Credential vaulting: stores privileged passwords in secure vault — rotates automatically
  • Session recording: records privileged sessions — accountability, forensics
Account Lifecycle
  • Provisioning: creating accounts with appropriate permissions when user joins
  • Deprovisioning: immediately removing all access when user leaves — critical after termination
  • Access recertification: periodic review of all access rights — removes stale/unnecessary permissions
  • SoD (Separation of Duties): no single person can both request and approve their own access
5.0 Security Program Management & Oversight 20%
⚖️ Security Governance & Compliance ▼
Governance is the organizational framework that ensures security aligns with business objectives and legal requirements. SY0-701 tests frameworks, regulations, and risk management extensively.
Security Frameworks
NIST CSF
National Institute of Standards and Technology Cybersecurity Framework — 5 functions: Identify, Protect, Detect, Respond, Recover. Most commonly referenced in US.
ISO 27001/27002
International standard for ISMS. 27001 defines requirements; 27002 provides implementation guidance for controls.
SOC 2
Trust Service Criteria audit — Type 1 (point in time) or Type 2 (period of time). Cloud providers often provide SOC 2 reports.
PCI-DSS
12 requirements for credit card data. Applies to any org storing, processing, or transmitting cardholder data.
HIPAA
Protects PHI. Security Rule requires administrative, physical, and technical safeguards. Breach notification required.
CMMC
Cybersecurity Maturity Model Certification — required for US DoD contractors. 3 maturity levels.
Risk Management Quantification
  • AV (Asset Value): monetary value of the asset
  • EF (Exposure Factor): percentage of asset lost if threat occurs (0–100%)
  • SLE (Single Loss Expectancy): AV × EF — cost of a single occurrence
  • ARO (Annual Rate of Occurrence): expected frequency per year
  • ALE (Annualized Loss Expectancy): SLE × ARO — expected annual loss — justifies security spend
💡
Example: Server worth $100,000. EF = 50% (half is lost if it fails). SLE = $50,000. ARO = 0.1 (fails once per 10 years). ALE = $5,000/year. If the control costs less than $5,000/year, it's cost-justified.
📋 Audits, Assessments & Awareness ▼
SY0-701 tests the ability to plan and interpret security assessments. Know the difference between vulnerability assessments, penetration tests, and audits.
Assessment Types
  • Vulnerability scan: automated — identifies known vulnerabilities — non-destructive — no exploitation
  • Penetration test: simulates real attack — active exploitation — requires written authorization
  • Black box pen test: tester has no prior knowledge — simulates external attacker
  • White box pen test: tester has full knowledge (source code, architecture) — simulates insider
  • Gray box pen test: partial knowledge — simulates partially informed attacker (credential access)
  • Red team: offensive — attacks systems to find weaknesses
  • Blue team: defensive — detects and responds to attacks
  • Purple team: collaborative — red and blue teams work together to improve overall security
Security Awareness Program Elements
  • Phishing simulations: send fake phishing emails — measure click rate, improve training targeting
  • Training programs: role-specific instruction — developers get secure coding, HR gets social engineering awareness
  • Gamification: point systems, leaderboards, rewards — increases engagement and retention
  • Tabletop exercises: discussion-based scenario walkthroughs — test plans without operational impact
  • Full-scale exercises: simulated incidents with real response — most accurate but most disruptive
  • AUP acknowledgment: users formally accept the acceptable use policy — legal and behavioral
